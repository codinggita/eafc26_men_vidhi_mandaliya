Documentation Overview

1. Introduction

Welcome to the **EAFC 26 Player Analytics API** documentation website. This service is built with Node.js and Express.js, providing highly optimized endpoints to search, filter, analyze, and aggregate player ratings, playstyles, and stats from the EAFC 26 Men's Football Dataset.

Whether you're building a squad generator, comparison dashboard, mobile player database, or team builder chemistry app, this REST API provides the backend calculations, pagination filters, role-based controls, and security headers required to build premium products.

Local Development Host: Make sure the local Express server is running on `http://localhost:5000` (or your chosen PORT) to make real API requests from this playground.

Key Features

  • MVC Architecture: Organized controller/model/route layout with Mongoose MongoDB mapping.
  • Interactive Testing (Try It Out): Execute API requests directly from this browser page. Toggle to Mock Mode to preview formatted responses even if the backend is local or offline.
  • Role-Based Security (RBAC): Endpoints are protected by JWT tokens with specific reader/writer permissions.
  • Aggregation Analytics: Live team ratings, player distribution graphs, age statistics, and distribution metrics.
  • Input Validation Layer: Joi validators verifying parameters, data shapes, and limits before executing queries.

Security

4. Authentication & Authorization

This API uses JSON Web Token (JWT) authorization to secure mutating actions and administrative statistics dashboards.

JWT Header Scheme

Include the token in the Authorization header using the standard Bearer scheme:

Authorization: Bearer
Permissions Configuration: The API defines specific permissions:
  • READ_ANALYTICS: Required for aggregating player ratings (Analytics APIs).
  • WRITE_PLAYERS: Required to perform CRUD inserts, updates, and deletes (POST/PUT/PATCH/DELETE /players).
  • ACCESS_DASHBOARD: Required for general stats and system logs (Admin APIs).
Traffic Guarding

Data Validation

7. Validation Rules

Incoming request body parameters and query structures are validated using Joi schemas. Failing to pass these constraints yields a 400 Bad Request status showing which validation parameters failed.

Sample Limits:

  • Overall Rating (OVR): Integer between 1 and 99.
  • Pace, Shooting, Passing, Dribbling, Defending, Physical: Integer between 1 and 99.
  • Age: Integer between 15 and 50.
  • Preferred Foot: Can only be "Left" or "Right".
  • Gender: Can only be "Men" or "Women".
Data Fetching Patterns

8. Pagination, Sorting & Filtering

Bulk queries support advanced pagination, query sorting, and parameter filters to paginate and format output data shapes.

Query Schema:

  • Pagination: Use page (defaults to 1) and limit (defaults to 10) params to paginate collections.
  • Sorting: Sort by attribute fields. Standard ascending format: ?sort=ovr. For descending ratings, append the desc tag: ?sort=ovr-desc (or -ovr depending on current model specifications).
  • Filtering: Exact value matches on team, league, nation, position, preferredFoot, and range constraints using prefix boundaries (e.g. minPace=90).
Parameters

9. Query & Route Parameters

We split parameter captures into Route Parameters (captured via Express path slugs) and Query Parameters (passed as standard URL arguments).

Route Parameter Syntax:

Route parameter values are embedded directly in the path (e.g. /players/231747, where 231747 is the ID route parameter).

Query Parameter Syntax:

Query parameters follow standard URI formatting (e.g. /players?team=Liverpool&minPace=85).

HTTP Responses

10. Status Codes Reference

The Player Analytics API communicates status details using standard HTTP codes:

Status Code Message Definition Typical Cause
200 OK Success Read or update query completed successfully.
201 Created Created Post request successfully registered records inside MongoDB.
400 Bad Request Bad Request Validation fails on headers or body params.
401 Unauthorized Unauthorized Missing or expired Bearer JWT authentication tokens.
403 Forbidden Forbidden Authenticated credentials lack permissions for action.
404 Not Found Not Found Requested ID or search term does not map to database records.
429 Too Many Requests Throttled Rate limits exceeded. System rejects requests temporarily.
500 Internal Server Server Error Unexpected backend exceptions caught by Express global handler.
Going Live

11. Deployment Guide

To deploy both the backend Express app and the documentation publicly, follow these general guidelines:

Express Backend App Deployment

The Express backend can be deployed to services like Render, Heroku, or digital ocean droplets. Since it relies on MongoDB, connect it to a managed MongoDB instance (like MongoDB Atlas) by updating the MONGO_URI env variable.

Static Documentation Deployment

This documentation page is a standalone client-side website and can be deployed instantly using Surge.sh, Netlify, or GitHub Pages. Run the automated deployment script included in the codebase to push updates to live staging immediately.
